# API Monitoring Platform

Платформа для регистрации внешних API/сервисов и непрерывного мониторинга их
доступности: фоновые async-воркеры опрашивают эндпоинты по расписанию,
пишут историю чеков в Postgres, считают аптайм/среднее время ответа и
кэшируют горячие данные в Redis.

## Стек

FastAPI · SQLAlchemy 2.0 (async) · PostgreSQL (asyncpg) · Alembic · Redis ·
httpx · Docker / docker-compose

## Возможности

- **Реестр сервисов** — CRUD (`/services`), категории, теги (Postgres `ARRAY`)
- **Фоновый мониторинг** — независимый от веб-процесса поллинг-шедулер,
  у каждого сервиса свой интервал проверки
- **Статистика** — `/services/{id}/stats`, `/services/{id}/history`,
  аптайм, среднее время ответа, последние инциденты
- **Health Dashboard** — `/dashboard` (внутренний) и `/status` (публичный,
  без внутренних id, зато с аптаймом)
- **Redis-кэш** — read-through на stats/dashboard/status с активной
  инвалидацией по записи нового чека (не просто TTL)
- **Rate limiting** — `slowapi`, лимит на IP
- **Telegram-алерт** — опционально, при переходе сервиса в `offline`
- **Распределённый лок мониторинга** — Redis `SET NX EX`, чтобы при
  нескольких репликах апи чеки не дублировались

## Запуск

```bash
cp .env.example .env
docker compose up --build
```

API будет на `http://localhost:8000`, документация — на `/docs`.

> Сам контейнерный билд в моей среде разработки не тестировался (нет
> доступа к Docker Hub из песочницы) — логика приложения целиком проверена
> живыми HTTP-запросами поверх локально поднятых Postgres/Redis процессов
> (миграции, CRUD, мониторинг настоящего `api.github.com`, кэш и его
> инвалидация, rate limit, лок — всё прогонялось руками). Перед тем как
> класть в резюме, стоит один раз самому прогнать `docker compose up`.

## API

| Метод | Путь | Что делает |
|---|---|---|
| POST | `/services` | создать сервис |
| GET | `/services` | список (+ category/tag/enabled_only фильтры) |
| GET | `/services/{id}` | один сервис |
| PATCH | `/services/{id}` | частичное обновление |
| DELETE | `/services/{id}` | удалить (каскадом чистит чеки и статистику) |
| GET | `/services/{id}/stats` | аптайм, среднее время, последние инциденты |
| GET | `/services/{id}/history` | сырая история чеков (limit, only_failures) |
| GET | `/dashboard` | health summary всех сервисов |
| GET | `/status` | публичная status-page |
| GET | `/health` | healthcheck апи |

## Структура

```
api-monitor/
├── main.py            # точка входа, lifespan, роутеры, rate limiter
├── models.py           # Service, ServiceCheck, ServiceStatistics
├── schemas.py           # все Pydantic-схемы
├── database.py           # singleton: async engine + session factory
├── core/
│   ├── settings.py        # конфиг из .env
│   ├── redis.py           # клиент redis, тот же lifecycle-паттерн
│   └── rate_limit.py       # slowapi limiter
├── modules/
│   ├── services.py        # CRUD реестра (роутер + логика)
│   ├── checker.py         # HTTP-чек внешнего сервиса, шаренный httpx-клиент
│   ├── stats.py           # /stats, /history
│   ├── dashboard.py       # /dashboard, /status
│   └── notifications.py    # telegram-алерт
├── tasks/
│   └── monitor.py         # поллинг-шедулер + запись результата
└── alembic/
```

Корневые `models.py`/`schemas.py`/`database.py` + `modules/<domain>.py` —
тот же подход, что в `backend-template`, адаптированный под то, что здесь
домены — это внешние API под мониторингом, а не пользователи: добавился
`modules/checker.py` (аналог `transfer.py` из бота) и `tasks/monitor.py`
вместо одной плоской бизнес-логики.

## Архитектурные решения

**1. Поллинг-шедулер вместо одной asyncio-задачи на сервис.**
Каждые 5 секунд шедулер спрашивает у БД, кто due (`last_check_at` старше
своего `check_interval_seconds`), и гоняет чеки `gather()`'ом. Альтернатива
— по `asyncio.Task` на сервис с собственным `while True: sleep(interval)` —
точнее по таймингу, но требует ручного управления жизненным циклом тасков
при создании/удалении/смене интервала сервиса через CRUD. Поллинг проще и
устойчивее к рестартам (due-логика всегда читает текущее состояние БД, а не
память процесса) ценой точности тайминга в пределах 5 секунд.

**2. `service_statistics` — инкрементальный агрегат, а не сканирование
`service_checks`.** total_checks/failed_checks — простые O(1) инкременты на
каждом чеке. `/stats` читается одной строкой без сканирования истории.

**3. `avg_response_time_ms` — индексированный агрегат, а не ручная
running-формула.** `AVG(...) WHERE service_id = ... AND response_time_ms IS
NOT NULL` по индексированной колонке — дешёво и без edge-кейсов с null'ами
на таймаутах. При реально больших объёмах истории это первое, что стоит
заменить на материализованный rollup (почасовые/посуточные агрегаты).

**4. `is_success = status_code < 500`, а не строго `2xx`.** Часть публичных
API намеренно отдают 4xx без авторизации/при рейт-лимите, оставаясь при
этом живыми — это видно на реальном `api.github.com` (403, но `online`).
Падением считаются только 5xx и сетевые сбои/таймауты.

**5. `current_status` — не просто "успешен последний чек".** offline —
последний чек зафейлился; degraded — успешен, но среди последних
`DEGRADED_LOOKBACK_CHECKS` есть фейлы (флапает) или время ответа выше
порога; иначе online. Пороги — в `core/settings.py`.

**6. Кэш инвалидируется явно, а не только по TTL.** После каждого записанного
чека `monitor.py` удаляет `stats:{id}`, `dashboard`, `status_page` из
Redis — следующий запрос гарантированно увидит свежие данные, TTL — только
подстраховка на случай, если инвалидация почему-то не сработала.

**7. DELETE и `is_enabled` — разные операции.** DELETE каскадом убирает
сервис и всю его историю насовсем. `is_enabled=false` ставит мониторинг на
паузу, не трогая накопленную историю — это разные сценарии ("выключили
интеграцию" vs "временно не мониторим").

**8. Лок на тик шедулера, а не полноценный leader election.** `SET NX EX
POLL_INTERVAL_SECONDS` в Redis — при нескольких репликах апи чеки не
задвоятся, лок сам протухает к следующему тику, не нужен heartbeat/renewal.
Цена — при падении лидера в худшем случае одна "пустая" попытка от другой
реплики, это нормально для мониторинга с интервалом в секунды-десятки
секунд.

## Чего нет (осознанно, не успело/не нужно для MVP)

- группировка инцидентов в диапазоны (сейчас `recent_incidents` — список
  отдельных зафейленных чеков, не "падение с 10:03 до 10:17")
- индивидуальный порог `degraded` на сервис (сейчас один глобальный)
- auth/API-ключи на запись (CRUD открыт всем — ок для портфолио-демо)
